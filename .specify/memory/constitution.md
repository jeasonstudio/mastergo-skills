# MasterGo Skills 项目宪章

## 核心理念

**清晰简洁，不做多余的事情。**

所有文档和代码必须清晰简洁。每一处新增都必须有明确的存在理由。

## 核心原则

### I. 代码质量

代码必须清晰、简洁、自解释。

- **单一职责**: 每个函数、模块、文件必须只有一个明确的用途
- **无死代码**: 立即删除未使用的代码；不要注释掉代码"留着以后用"
- **最小依赖**: 仅当收益明显大于成本时才添加外部依赖
- **自文档化**: 代码结构和命名必须传达意图；注释解释"为什么"而非"是什么"
- **风格一致**: 遵循既有模式；仅在有明确理由时引入新模式

**理由**: 清晰的代码降低认知负担，加速调试，使修改更有信心。

### II. 测试标准

测试必须验证行为，而非实现。

- **聚焦测试**: 每个测试必须验证一个特定的行为或场景
- **无冗余测试**: 不要编写多个测试来覆盖相同的代码路径
- **测试重点**: 优先为 MCP 工具交互编写集成测试；为工具函数编写单元测试
- **清晰断言**: 测试名称和断言必须清晰描述预期行为
- **可维护**: 重构实现细节时，测试不应需要更新

**理由**: 聚焦的测试能捕获真实 bug，而不会造成维护负担。

### III. 用户体验一致性

Skill 行为必须可预测且一致。

- **一致的 API**: MCP 工具在所有调用中必须返回一致的数据结构
- **可预测的错误**: 错误消息必须清晰说明问题所在及解决方法
- **稳定的契约**: 工具签名的破坏性变更需要主版本号升级
- **文档准确性**: 文档必须反映实际行为；代码变更时同步更新文档
- **渐进式披露**: 优先展示关键信息；按需提供详细内容

**理由**: 一致的行为建立用户信任，减少支持负担。

### IV. 性能要求

操作必须高效完成，资源使用最小化。

- **快速响应**: MCP 工具调用对于典型操作应在 5 秒内完成
- **最小内存**: 当部分遍历足够时，避免将整个 DSL 树加载到内存
- **无多余请求**: 适当缓存 MasterGo API 响应；避免重复调用
- **延迟加载**: 仅在需要时加载组件链接和嵌套数据
- **快速失败**: 尽早验证输入；不要处理无效数据

**理由**: 高效的操作确保工作流顺畅集成。

### V. 语言规范

内容语言必须根据目录位置确定。

- **中文目录**: `.cursor/`、`.specify/`、`/specs` 目录内的内容必须使用中文
- **英文目录**: 其他所有目录的内容必须使用英文
- **代码注释**: 代码注释遵循所在目录的语言规范
- **提交信息**: Git 提交信息使用英文

**理由**: 统一的语言规范确保团队协作效率，配置与规范使用中文便于理解，代码使用英文保持国际兼容性。

## 质量关卡

### 代码审查清单

合并任何变更前：

- [ ] 代码有明确、文档化的用途
- [ ] 无死代码或注释掉的代码块
- [ ] 测试验证行为而非实现
- [ ] 错误处理明确且有帮助
- [ ] 性能影响可接受
- [ ] 语言规范正确（中文/英文目录）

### 测试关卡

- **脚本**: 所有 JavaScript 工具必须在注释中包含使用示例
- **工作流**: 参考文档必须包含可运行的示例
- **MCP 集成**: 工具交互必须用真实 MasterGo 数据测试

## 开发流程

### 变更流程

1. **识别需求**: 这解决什么问题？是否必要？
2. **最小方案**: 满足需求的最简单变更是什么？
3. **实现**: 编写清晰的代码；更新相关文档
4. **验证**: 测试变更；确保无回归
5. **审查**: 对照质量关卡检查

### 文件组织

- `SKILL.md`: 入口点；保持简洁，链接到参考文档
- `references/`: 详细的工作流和数据结构
- `scripts/`: 具有明确单一用途的可复用工具

## 治理

本宪章定义了 MasterGo Skills 项目的不可协商标准。

### 修订流程

1. 提出变更并说明理由
2. 评估对现有代码和文档的影响
3. 更新所有受影响的制品
4. 适当递增版本号

### 版本策略

- **MAJOR**: 原则的破坏性变更或需求移除
- **MINOR**: 新增原则或扩展指导
- **PATCH**: 澄清、措辞改进、错别字修复

### 合规性

所有贡献必须遵守本宪章。不合规的代码应当重构。
